# HIS 项目开发规则

## 项目名称
社区智慧医疗信息管理系统（HIS）

## 项目背景
这是一个课程设计项目，目标是基于链表实现一个社区智慧医疗信息管理系统。
当前项目已经有主程序入口、全局结构体、工具函数、底层双向链表操作。
请在现有代码基础上继续开发，不要推翻重写。

## 当前已有文件职责
- main.c：系统主入口与主菜单
- global.h：全局常量、枚举、核心结构体、全局链表头指针声明
- utils.h / utils.c：安全输入工具函数
- list_ops.h / list_ops.c：底层双向链表的初始化、创建、插入、查找、删除等操作

## 当前开发阶段
当前处于“底层骨架已完成，业务模块待接入”的阶段。
优先补齐数据模型，并实现第一个完整业务闭环，不要一次性扩展整个系统。

## 第一优先级任务
1. 补齐患者与预约相关的数据结构
2. 实现患者建档
3. 实现预约链表管理
4. 实现预约查询、取消、签到
5. 将 main.c 中的“快捷挂号”和“患者自助”接入最小可运行菜单

## 暂时不要做的内容
在没有明确要求前，不要扩展以下模块：
- 检查检验管理
- 电子处方收费
- 药房库存联动
- 住院与床位日结
- 文件存储与数据持久化
- 加密与复杂日志系统
- 回收站完整实现
- 数据大屏完整实现

## 业务规则总原则
所有修改必须以题签设计书为准，不要擅自改核心规则。

### 患者相关规则
1. 患者建档时需要录入姓名、年龄、身份证号
2. 患者编号自动生成，例如 P-001
3. 身份证号必须做格式校验
4. 身份证号展示时应支持脱敏
5. 同一个身份证号不允许重复建档

### 预约相关规则
1. 系统需要新增 AppointmentNode，并使用双向链表管理预约记录
2. 预约编号自动生成，例如 A-001
3. 预约记录至少包含：
   - appointment_id
   - patient_id
   - appointment_date
   - appointment_slot
   - appoint_doctor 或 appoint_dept
   - appointment_status
4. 预约状态至少支持：
   - RESERVED（已预约）
   - CHECKED_IN（已签到）
   - CANCELLED（已取消）
   - MISSED（已过号）
5. 仅未签到预约允许取消
6. 已签到和已过号预约不允许物理删除，只能修改状态
7. 预约签到后，应将患者转入待诊状态
8. 如果预约指定了医生，则签到后该医生 queue_length 加 1

### 病历与隐私相关规则
1. 患者查阅病历时必须通过患者编号与身份证号联合核验
2. 不能只凭患者编号直接开放病历
3. 身份证号属于敏感字段，输出时尽量脱敏

## 数据结构修改要求
1. 在 global.h 中优先补充和修正：
   - PatientNode 增加 id_card 字段
   - 新增 AppointmentStatus 枚举
   - 新增 AppointmentNode 结构体
   - 新增全局头指针 g_appointment_list
2. 保持现有命名风格，不要无故大改已有结构
3. 新增结构尽量和现有双向链表风格保持一致
4. 尽量使用已有宏，例如 MAX_ID_LEN、MAX_NAME_LEN

## 工具函数要求
在 utils.h / utils.c 中，除保留现有安全输入函数外，可新增：
- validate_id_card
- mask_id_card

要求：
1. validate_id_card 先实现基础格式校验
2. mask_id_card 用于脱敏显示身份证号
3. 所有新菜单输入尽量调用 get_safe_int、get_safe_double、get_safe_string

## 底层代码修正规则
修改 list_ops.h / list_ops.c 时：
1. 只做必要修正，不要重构成完全不同的风格
2. 修正声明与实现不一致的问题
3. 新增 AppointmentNode 对应的底层操作时，延续当前风格：
   - init_xxx_list
   - create_xxx_node
   - insert_xxx_tail
   - find_xxx_by_id
4. 删除、查找、插入函数尽量保持接口风格统一
5. 注意字符串长度边界，避免 strncpy 越界或命名不一致

## main.c 修改规则
1. 保留当前主菜单整体样式
2. 优先接入：
   - 选项 2：快捷挂号/患者建档/预约管理
   - 选项 3：患者自助预约查询
3. 先不要接入完整登录鉴权
4. 先不要接入完整医疗大屏
5. 新增菜单时保持控制台课程设计风格，简洁清晰即可

## 代码风格要求
1. 保持 C 语言课程设计风格，避免过度工程化
2. 优先写清晰、直白、便于老师检查的代码
3. 注释可以保留当前风格，但不要过度花哨
4. 不要引入复杂第三方库
5. 不要擅自改成 C++ 风格
6. 不要一次性创建大量无关文件

## 每次任务的输出要求

1. 修改了哪些文件
2. 每个文件改了什么
3. 为什么这样改
4. 当前代码能否直接编译
5. 还缺什么下一步才能继续

4. 不要提前扩展到完整 HIS 全流程
5. 当前只围绕“患者 + 预约 + 身份核验”优先开发
